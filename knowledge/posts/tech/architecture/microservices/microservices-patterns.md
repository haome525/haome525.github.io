# 微服务架构核心模式

微服务架构通过将系统拆分为独立部署的小服务来提高灵活性和可维护性。

## 1. 服务发现

服务实例动态注册与发现：

- **客户端发现**：客户端从注册中心获取实例列表（Eureka）
- **服务端发现**：通过负载均衡器转发请求（Kubernetes Service）

## 2. API 网关

统一入口，负责路由、认证、限流、聚合：

```
客户端 → API 网关 → 微服务 A
                    → 微服务 B
                    → 微服务 C
```

## 3. 配置中心

集中管理多环境的配置：

```yaml
# application.yml
server:
  port: 8080
spring:
  config:
    import: configserver:http://config-server:8888
```

## 4. 容错模式

| 模式 | 描述 |
|------|------|
| 断路器（Circuit Breaker） | 失败快速返回，避免级联故障 |
| 隔板（Bulkhead） | 隔离资源池，一个服务故障不影响其他 |
| 重试（Retry） | 对临时故障进行自动重试 |
| 超时（Timeout） | 防止请求无限等待 |

## 5. 分布式追踪

使用 OpenTelemetry 或 Jaeger 追踪跨服务调用链。

## 6. 挑战

- **数据一致性**：最终一致性 vs 强一致性权衡
- **调试复杂度**：跨服务调用链难以本地复现
- **部署复杂度**：CI/CD 和容器编排不可或缺
